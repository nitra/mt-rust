## Огляд

`layers.json` — єдине джерело даних для конфігурації шарів документації; код спирається на нього, щоб отримувати структуру шарів, межі перекладу та мовні шляхи документів.

`CONFIG_NAME="layers.json"` — назва конфігураційного файлу, який використовує модуль.

Публічні функції `loadConfig`, `validateConfig`, `topoOrder`, `translationScope`, `langPath`, `layerNumber` працюють із цим конфігом і разом формують перевірену модель шарів для читання та навігації по документації.

## Поведінка

- CONFIG_NAME — назва конфігураційного файла шарової документації: `layers.json`.
- loadConfig — читає `layers.json`, збирає конфіг топології документації, перевіряє його валідність і повертає готову структуру для подальшої обробки.
- layerNumber — перетворює мітку шару на порядковий номер, щоб порівнювати шари між собою.
- validateConfig — перевіряє узгодженість шарів, напрямок посилань між доками, коректність режимів і заборону використовувати fragment-доки як джерела.
- topoOrder — повертає порядок перебудови документів з нижчих шарів до вищих, зберігаючи порядок оголошення всередині шару.
- translationScope — повертає повний набір файлів, які мають потрапити в переклад: leaf-доки та всі доки з конфігу.
- langPath — будує шлях до мовної версії документа шляхом додавання коду мови до назви файла.

## Публічний API

- CONFIG_NAME — ім’я файлу з топологією шарів, яке код читає як стандартний конфіг.
- loadConfig — читає та звіряє конфіг топології з `<docsDir>/layers.json`.
- layerNumber — дістає номер шару з мітки `L<n>`; менше `n` означає вищий, більш агрегований шар.
- validateConfig — звіряє мітки шарів, напрямок джерел лише з нижчих шарів угору та обмеження fragment; така побудова прибирає цикли.
- topoOrder — задає порядок перебудови знизу вгору, а всередині шару тримає порядок із конфігу.
- translationScope — визначає набір файлів для перекладу: leaf-доки та всі документи з конфігу.
- langPath — формує шлях derived-перекладу, наприклад `architecture/graph.md` + `en` → `architecture/graph.en.md`.
